# 最小可行产品（MVP）

> 一堂五步法 · 第二步：解决方案
> 修炼水平：L3（从有价值意识 → 有最小意识）

---

## 1. MVP 核心假设

MVP 的目标不是"做最多功能"，而是**用最小成本验证核心假设**。以下四个假设覆盖了 OTTO 产品价值链的关键环节：学生兴趣、教师易用性、机构商业价值、教学效果可衡量性。

| # | 核心假设 | 验证指标 | 验证方式 |
|---|----------|----------|----------|
| 1 | 初中生对 AI 语音机器人有持续兴趣 | 完课率 ≥ 80% | 平台使用数据 + 考勤记录 |
| 2 | 教师可以零培训上手使用 | 教师满意度 ≥ 80% | 问卷调查 + 课堂观察 |
| 3 | 机构引入后出勤率/续课率有可见提升 | 运营数据对比 | 引入前后 A/B 对比 |
| 4 | 学生 2 节课后能独立编程、8 节课后能完成作品 | 作品提交数据 | 平台提交记录 + 作品评审 |

**验证逻辑**：四个假设构成递进关系——学生有兴趣（假设1）是前提，教师能教（假设2）是保障，机构有价值（假设3）是商业基础，教学有成果（假设4）是长期粘性。

## 2. MVP 范围定义

PRD 定义了 33 项需求（R1-R33），MVP 阶段按三级优先级筛选。

### 2.1 P0 必须（MVP 核心）

P0 模块覆盖了"硬件可用 → 软件可编程 → 课堂可交付 → 机构可管理"的完整闭环。

| 模块 | 需求编号 | 交付物 | 说明 |
|------|----------|--------|------|
| 硬件 | — | OTTO 套件 v1 | ESP32-S3 + 8MB PSRAM + ES8311，≤250 元/台 |
| 固件 | R1-R5 | 语音交互核心 | 离线唤醒（ESP-SR）+ 在线 AI 对话 + 语音控制动作 + OTA |
| 配置工具 | R6-R9 | 网页端配置 | 可视化参数调整 + 舵机校准 + OTA 升级 + 多端适配 |
| 编程工具 | R14-R17 | 图形化编程 | 28+ 预设积木 + 3D 模拟预览 + 12 节内置课程 |
| 竞赛 | R20 | 机构内部赛 | 外观选美 + 动作展示（手动组/AI 组）+ 功能创新赛 |
| 权限 | R24-R27 | 三级权限后台 | 管理员/教师/家长 + 内容推送 + 安全审计 + 使用监控 |
| 课堂 | R28-R30 | 多人共用 | 用户切换 + 课堂模式 + 紧急停止 |

### 2.2 P1 应该（MVP 增强）

P1 模块能显著提升体验，但核心价值验证不依赖它们。如果资源允许，在 MVP 试点后期加入。

| 模块 | 需求编号 | 交付物 | 说明 |
|------|----------|--------|------|
| DIY 扩展 | R10-R13 | 外壳 + 扩展模块 | 3D 打印外壳 + 摄像头/显示屏/额外舵机 |
| 竞赛增强 | R23 | 评分模板 | 教师可自定义评分维度和权重 |
| 设备管理 | R31-R33 | 设备运维 | 状态监控 + 故障诊断 + 备用设备策略 |

### 2.3 P2 延后（Phase 2+）

P2 模块需要更大的用户基数或更成熟的基础设施支撑，MVP 阶段明确不纳入。

| 模块 | 需求编号 | 交付物 | 延后理由 |
|------|----------|--------|----------|
| AI 创作 | R18 | AI 辅助动作生成 | 需 500+ 训练样本，MVP 不需要 |
| 音乐配合 | R19 | 动作音乐同步 | 非核心教学功能 |
| 机构联赛 | R21 | 机构间竞赛 | 需先验证单机构内部赛 |
| 线上社区 | R22 | 作品社区 | 需足够用户基数 |

## 3. MVP 成功标准

成功标准来源于 PRD 第 5 节，按验证周期排列，确保试点过程中能持续收集反馈。

| 指标 | 标准 | 验证方式 | 验证周期 |
|------|------|----------|----------|
| 学习曲线 | 学生 2 节课后独立完成简单动作编程（≥3 个积木块） | 课堂观察记录 | 试点前 4 周 |
| 作品产出 | 学生 8 节课后完成自定义外壳 + 编程动作组合 | 作品提交数据 | 试点前 8 周 |
| 机构价值 | 引入后出勤率和续课率有可见提升 | 机构运营数据对比 | 试点 1 学期 |
| 差异化 | 机构反馈"机器人课程"成为招生亮点 | 管理者访谈 | 试点 1 学期 |
| 系统可靠性 | 课堂中断 < 1 次/月/机构 | 故障日志统计 | 持续监测 |
| 用户满意度 | 教师满意度 ≥ 80%，学生满意度 ≥ 85% | 问卷调查 | 试点中期 |

**关键判定规则**：

- **硬性门槛**：系统可靠性必须达标，否则课堂无法正常运转。
- **核心指标**：学习曲线和作品产出是教学价值的直接体现。
- **商业指标**：机构价值和差异化是商业模型验证的关键。
- **软性指标**：用户满意度作为参考，用于发现改进方向。

## 4. MVP 验证方案

### 4.1 试点设计

| 维度 | 方案 |
|------|------|
| 试点规模 | 2-3 家机构（晚托班 / 学校社团课） |
| 试点周期 | 1 个学期（约 16 周） |
| 覆盖学生 | 100+ 名学生 |
| 设备投入 | 每机构 10 台（支持 20-30 名学生） |
| 验证方法 | A/B 对比（引入 OTTO 前后的出勤率、续课率、满意度对比） |

### 4.2 数据采集

| 数据类型 | 采集方式 | 采集频率 |
|----------|----------|----------|
| 平台使用数据 | 自动埋点（登录次数、编程时长、作品提交） | 实时 |
| 出勤率/续课率 | 机构运营系统对接或手动录入 | 每周 |
| 教师满意度 | 问卷（5 分制量表 + 开放性问题） | 第 4、8、12、16 周 |
| 学生满意度 | 问卷（简化版，课堂中填写） | 第 4、8、16 周 |
| 管理者反馈 | 半结构化访谈 | 第 8、16 周 |
| 作品质量 | 作品提交记录 + 教师评分 | 每节课 |
| 故障日志 | 系统自动记录 + 教师反馈 | 实时 |

### 4.3 Go/No-Go 决策点

| 时间节点 | 决策内容 | 判定依据 |
|----------|----------|----------|
| 第 4 周 | 是否继续试点 | 学习曲线达标 + 无重大技术故障 |
| 第 8 周 | 是否扩大范围 | 作品产出达标 + 教师满意度 ≥ 70% |
| 第 16 周 | 是否进入 Phase 2 | 机构价值验证通过 + 核心指标全部达标 |

## 5. 砍什么，为什么砍

MVP 的本质是做减法。以下逐项说明 P2 功能砍掉的理由，以及未来重新纳入的条件。

### R18 AI 辅助动作生成

**砍掉理由**：AI 动作生成需要标注 500+ 训练样本（不同动作、不同速度、不同体型外壳），数据采集和标注成本高。MVP 阶段手动图形化编程已能充分验证教学价值——学生通过拖拽积木理解编程逻辑，这正是课程的核心目标。

**重新纳入条件**：当试点机构积累 200+ 条学生编写的动作序列后，可作为训练数据启动 AI 动作生成。

### R19 动作音乐同步

**砍掉理由**：音乐配合属于"锦上添花"功能，增加技术复杂度（音频时序同步、版权处理）但不影响核心价值验证。学生的成就感主要来自"让机器人动起来"和"完成自己的作品"，音乐同步是后续的体验优化。

**重新纳入条件**：Phase 2 竞赛体系完善后，音乐同步可作为竞赛表演的加分项引入。

### R21 机构间联赛

**砍掉理由**：联赛需要先在单机构内验证竞赛模式的参与度、公平性和运营可行性。过早引入跨机构竞赛会分散试点精力，且需要额外的赛事组织能力。

**重新纳入条件**：至少 3 家机构完成内部赛且参与度达标后，再设计联赛规则和赛程。

### R22 线上社区

**砍掉理由**：社区需要足够的用户基数才能产生网络效应（作品展示 → 互相激励 → 持续创作）。MVP 阶段 100+ 名学生分散在 2-3 家机构，不足以支撑活跃的社区氛围。

**重新纳入条件**：累计覆盖 10+ 机构、500+ 学生，且作品提交率达到一定水平后再启动社区功能。

## 6. 最小可复制单元

从业务单元模型中提取"晚托植入点"作为最小可复制单元，验证单点盈利能力。

```
晚托植入点（最小可复制单元）：
├── 首期投入：¥3,300（10 台入门套件 × ¥330）
├── 月收入：¥8,000（20 人 × ¥400/月）
├── 月成本：¥3,700（师资 ¥2,140 + 固定 ¥1,560）
├── 月毛利：¥4,300
├── 毛利率：54%
├── 回本周期：约 1.4 个月
└── 运营复杂度：低（无需自有场地）
```

**复制逻辑**：晚托植入点的优势在于"轻资产"——不需要自有场地，借助合作机构现成的教室和学生资源，设备成本即可在 1-2 个月回收。这是 MVP 验证的首选场景。

**复制条件**：每个最小可复制单元需要满足：
1. 合作机构有独立的教室空间（可容纳 10 台设备）
2. 机构提供稳定的 5GHz Wi-Fi 网络
3. 机构有可安排的授课时段（每周至少 2 次，每次 1.5 小时）
4. 机构能组织 15-25 名学生参与课程

## 7. MVP 技术约束

以下约束来自 PRD 定义的技术规格，是 MVP 实现的边界条件。

| 维度 | 约束 | 说明 |
|------|------|------|
| 硬件平台 | ESP32-S3 + 8MB PSRAM + ES8311 | 主控 + 内存 + 音频芯片，单台成本 ≤250 元 |
| 网络环境 | 独立 5GHz Wi-Fi AP | 单课堂支持 15-20 台设备同时在线 |
| 通信协议 | MQTT（控制通道）+ WebSocket（媒体/音频流） | 控制与媒体分离，降低延迟 |
| 浏览器兼容 | Chrome ≥ 90、Safari ≥ 15、Edge ≥ 90 | 覆盖主流浏览器 |
| 并发规模 | 10 个机构（150-200 台设备）同时在线 | MVP 阶段的目标容量 |
| 传输加密 | TLS 1.2+ | 所有网络通信强制加密 |
| 存储加密 | AES-256 | 用户数据和配置信息加密存储 |
| 课堂配比 | 2-3 名学生共用 1 台机器人 | 影响用户切换和课堂管理设计 |

**技术风险提示**：

- ESP32-S3 的 8MB PSRAM 是硬性需求，原生 SRAM 无法同时运行语音唤醒 + 音频编解码 + 舵机控制 + Wi-Fi
- 课堂并发 15-20 台设备对 Wi-Fi AP 的性能有较高要求，需确保独立 AP 且无干扰
- OTA 升级采用分批策略（每批 3-5 台），避免同时升级导致网络拥塞
